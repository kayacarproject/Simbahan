Section 4: Android Signing — Key Components

4.1 Keystore Components Explained

Component Description
Keystore File (.jks) A binary file that stores one or more cryptographic key entries. This file signs the application and proves the identity of the publisher. Must be kept in a secure, backed-up location. File for this project: simbahan-release.jks
Key Alias A unique string identifier for the specific key entry inside the keystore. A single keystore file can contain multiple aliases. Example: simbahan-key
Keystore Password The password that protects access to the entire keystore file. Required every time the file is used to sign a build.
Key Password A separate password that protects the specific key entry identified by the alias. May be the same as the keystore password but is stored and managed independently.
Validity Period Google Play requires the signing certificate to remain valid past October 22, 2033. Recommended: set validity to 10,000 days (approximately 27 years) when generating the keystore.
SHA-1 Fingerprint A hash of the certificate, required to register the app with Google APIs such as Firebase, Google Maps, and OAuth. Generated from the keystore using the keytool command.

4.2 Retrieving the SHA-1 Fingerprint

Run the following command, replacing the values with your actual keystore details:

keytool -list -v \
  -keystore android/app/simbahan-release.jks \
  -alias simbahan-key

The output will include the SHA-1, SHA-256, and MD5 fingerprints.

Section 5: Steps to Generate a Release .aab File

Follow the steps below in sequence. All commands are run from the terminal at the project root unless otherwise stated.

Step 1 — Generate the Keystore File

Run the following command from your terminal. Replace all placeholder values with your project-specific information. You will be prompted to enter a keystore password, key password, and owner information.

keytool -genkeypair -v \
  -keystore android/app/simbahan-release.jks \
  -alias simbahan-key \
  -keyalg RSA \
  -keysize 2048 \
  -validity 10000

When prompted, provide:

  • First and last name (or organization name)
  • Organizational unit
  • Organization
  • City, State, Country code (e.g., PH for Philippines)
Step 2 — Place the Keystore File

Move or confirm the generated keystore file is located at:

android/app/simbahan-release.jks

Do not place the keystore outside the android/app/ directory, as the build script references it relative to this path.

Step 3 — Configure gradle.properties

Open the file android/gradle.properties and add the following lines at the bottom. Replace the values with your actual credentials.

MYAPP_UPLOAD_STORE_FILE=simbahan-release.jks
MYAPP_UPLOAD_KEY_ALIAS=simbahan-key
MYAPP_UPLOAD_STORE_PASSWORD=your_keystore_password
MYAPP_UPLOAD_KEY_PASSWORD=your_key_password
Security Note: Do not commit gradle.properties to any public repository if it contains credentials. Add it to .gitignore or use environment variables in a CI/CD pipeline.
Step 4 — Update android/app/build.gradle

Open android/app/build.gradle. Inside the android { } block, add or update the signingConfigs and buildTypes sections as shown below.

android {
    signingConfigs {
        release {
            storeFile file(MYAPP_UPLOAD_STORE_FILE)
            storePassword MYAPP_UPLOAD_STORE_PASSWORD
            keyAlias MYAPP_UPLOAD_KEY_ALIAS
            keyPassword MYAPP_UPLOAD_KEY_PASSWORD
        }
    }
    buildTypes {
        release {
            signingConfig signingConfigs.release
            minifyEnabled false
            shrinkResources false
            proguardFiles getDefaultProguardFile("proguard-android.txt"),
                          "proguard-rules.pro"
        }
    }
}
Step 5 — Clean the Project

Navigate into the android directory and run the Gradle clean task to remove any previous build artifacts.

cd android
./gradlew clean
Step 6 — Build the Release Bundle

While still inside the android directory, run the release bundle task.

./gradlew bundleRelease

The build process will compile the JavaScript bundle, apply Hermes compilation, sign the output, and produce the final .aab file. This may take several minutes on first run.

Step 7 — Locate the Output File

After a successful build, the signed .aab file is located at:

android/app/build/outputs/bundle/release/app-release.aab

This file is ready for upload to Google Play Console under any of the following tracks:

  • Internal Testing
  • Closed Testing (Alpha)
  • Open Testing (Beta)
  • Production

Section 6: Privacy Policy Requirements

A Privacy Policy is mandatory on both Google Play and the Apple App Store. The following table details all requirements.

Requirement Details
HostingMust be hosted on a publicly accessible, live URL. A PDF link is not accepted.
FormatStandard web page (HTML). Must be accessible without login.
Data Collection DisclosureMust describe what data is collected (name, email, device data, usage data, etc.), how it is used, and how long it is retained.
Third-Party ServicesMust disclose all third-party SDKs and services used (e.g., Firebase, Google Analytics, push notification services).
Permissions DisclosureMust explain the purpose of every permission requested by the app: camera, microphone, location, notifications, and any other sensitive permissions.
User RightsMust describe how users can request access to, correction of, or deletion of their data.
Contact InformationMust include a valid email address for privacy-related inquiries.
GDPR / COPPAIf the app is available in the European Union or to users under 13, additional GDPR and COPPA compliance language is required.
UpdatesMust be updated whenever the app's data collection or usage practices change.
Recommended Tools: Use App Privacy Policy Generator (app-privacy-policy-generator.firebaseapp.com) or Termly (termly.io) to generate a compliant starting document. Always review and customise the output to match actual app behaviour.
